## 1. 项目概述

### 1.1 系统简介

本项目是一个**多 RTU 物联网数据采集网关系统**，采用 Maven 多模块架构。系统从串口设备以 1 秒间隔采集温湿度数据，通过 RTU 网关使用 Netty 处理，并通过 MQTT 转发到 Web 服务器进行管理和可视化展示。

### 1.2 核心特性

- **分布式部署**：边缘采集与中心处理分离
- **高频采集**：1 秒间隔实时数据采集
- **可靠传输**：断网缓存与自动重连机制
- **纯字节流通信**：serial-collector ↔ rtu-gateway 使用自定义二进制协议，无 JSON 依赖
- **一对一架构**：一个 RTU 对应一个串口，一个串口对应一个 Modbus 设备（串口点对点通信）
- **集中管理**：统一的 RTU 网关注册、配置和监控
- **灵活配置**：支持远程下发采集参数和报警阈值

## 2. 整体架构

### 2.1 核心概念说明

**重要**：理解以下概念对架构设计至关重要：

- **rtuId**：代表一个**边缘采集设备**（如工控机、树莓派）
  - 示例：RTU-001 代表"1号车间的边缘采集设备"
  - **一个 RTU 对应一个串口，一个串口对应一个 Modbus 设备**（串口是点对点通信）
  - rtuId 在整个系统中全局唯一

- **deviceId**：与 rtuId 相同，代表唯一的温湿度传感器设备
  - 由于串口是一对一通信，deviceId = rtuId
  - 示例：RTU-001 既是边缘设备标识，也是传感器设备标识

- **serial-collector**：运行在边缘设备上的采集程序
  - 每个 serial-collector 实例拥有一个 rtuId
  - 负责 Modbus-RTU 通信和数据采集
  - 管理一个串口，连接一个 Modbus 设备（固定地址，通常为 0x01）
  - 作为 TCP 客户端连接到中心服务器的 rtu-gateway
  - **使用纯二进制协议通信，不涉及 JSON**

- **rtu-gateway**：运行在中心服务器的**协议转换服务**
  - 不拥有 rtuId，负责协议转换（二进制 ↔ JSON）
  - 单进程接受多个 serial-collector 连接
  - 根据二进制认证帧识别不同的 rtuId
  - 从 Modbus 报文提取设备地址（通常固定为 0x01）

- **认证密钥（secret）**：每个 rtuId 对应一个密钥
  - 在 web-server 注册 RTU 时生成
  - serial-collector 连接时通过二进制帧发送 rtuId + secret（SHA-256加密）
  - rtu-gateway 调用 web-server API 验证

- **通信协议**：自定义二进制帧格式
  - 帧头：0xAA55（固定，用于帧同步）
  - 帧类型：认证(0x01)、数据(0x03)、命令(0x04)、心跳(0x05)等
  - 数据体：Modbus 原始报文或认证信息
  - 校验和：异或校验

### 2.2 系统分层架构

系统采用四层架构设计：

1. **边缘采集层**：serial-collector 运行在边缘设备（工控机/树莓派），通过 Modbus-RTU 协议读取温湿度传感器，每秒采集一次数据
2. **网关接入层**：rtu-gateway 运行在中心服务器，作为无状态的 Netty TCP 服务端，接收多个 serial-collector 的连接并转发到 MQTT
3. **消息中间件层**：独立部署的 Mosquitto MQTT Broker，负责消息路由和分发
4. **业务服务层**：web-server（Spring Boot）提供 RTU 注册管理、认证验证、数据存储、报警处理、RESTful API 和前端服务

### 2.3 物理部署拓扑

```
[边缘设备1 - RTU-001]              [中心服务器]
  串口 /dev/ttyUSB0
    └── Modbus设备 (地址 0x01)      ┌─────────────────┐
      ↓ 串口 Modbus-RTU            │  rtu-gateway    │
  serial-collector ───二进制帧────→│  (Port 8888)    │
  (rtuId=RTU-001)                 │  - Netty Server │
  (secret=abc123)                 │  - 协议转换      │
                                  │  - 帧解析器      │
[边缘设备2 - RTU-002]              └────┬────────────┘
  串口 /dev/ttyUSB0                    │ MQTT (JSON)
    └── Modbus设备 (地址 0x01)          ↓
      ↓ 串口 Modbus-RTU         ┌─────────────────┐
  serial-collector ───二进制帧──→│   Mosquitto     │
  (rtuId=RTU-002)               │  MQTT Broker    │
  (secret=xyz789)               │  (Port 1883)    │
                                └────┬────────────┘
[边缘设备N - RTU-00N]                 │ MQTT (JSON)
  串口 /dev/ttyUSB0                   ↓
    └── Modbus设备 (地址 0x01)  ┌─────────────────┐
      ↓ 串口 Modbus-RTU        │   web-server    │
  serial-collector ───二进制帧→ │  - RTU 管理      │
  (rtuId=RTU-00N)              │  - 认证验证 API  │
  (secret=def456)              │  - MQTT Client  │
                               │  - MySQL        │
                               │  - REST API     │
                               │  - Frontend     │
                               └─────────────────┘
```

**说明**：
- 每个边缘设备有唯一的 rtuId（如 RTU-001）
- **一个 RTU 对应一个串口，一个串口对应一个 Modbus 设备**（串口是点对点通信）
- deviceId = rtuId（设备标识与 RTU 标识相同）
- Modbus 设备地址通常固定为 0x01
- serial-collector ↔ rtu-gateway 使用自定义二进制帧协议
- rtu-gateway ↔ MQTT/web-server 使用 JSON 格式
- rtu-gateway 负责协议转换（二进制 ↔ JSON）
- 所有 RTU 的注册信息、密钥、状态都存储在 web-server 的数据库中

### 2.4 数据流向

#### 上行数据流（数据采集）
```
温湿度传感器 (Modbus地址 0x01)
  ↓ (Modbus-RTU 串口通信)
serial-collector (rtuId=RTU-001, 读取寄存器0x0000-0x0001)
  ↓ 封装为二进制帧 (帧类型0x03)
  [AA 55 03 00 09 01 03 04 02 92 FF 9B 5A 3D XX]
  ↓ (Netty TCP，二进制帧)
rtu-gateway (解析二进制帧，提取 Modbus 报文)
  ↓ 从 channel 识别 rtuId = RTU-001
  ↓ 从 Modbus 报文提取设备地址 = 0x01（固定）
  ↓ deviceId = rtuId = RTU-001
  ↓ 解析温湿度数据
  ↓ (MQTT发布到 iot/rtu/RTU-001/data，JSON格式)
MQTT Broker
  ↓ (MQTT订阅 iot/rtu/+/data)
web-server (从 topic 提取 rtuId，验证 RTU 状态，存储数据，报警判断)
  ↓ (REST API，JSON格式)
前端展示 (按 RTU 展示数据)
```

#### 下行数据流（配置下发）
```
前端操作 (选择 RTU-001，修改配置)
  ↓ (HTTP POST /api/rtu/RTU-001/config，JSON格式)
web-server (保存配置到数据库 rtu_config 表)
  ↓ (MQTT发布到 iot/rtu/RTU-001/config，JSON格式)
MQTT Broker
  ↓ (MQTT订阅 iot/rtu/RTU-001/config)
rtu-gateway (接收 JSON 配置，转换为 Modbus 写入命令)
  ↓ 封装为二进制帧 (帧类型0x04)
  [AA 55 04 00 08 01 06 07 D1 00 02 88 3A XX]
  ↓ (Netty TCP，二进制帧)
serial-collector (解析二进制帧，提取 Modbus 命令)
  ↓ (Modbus-RTU 0x06写寄存器)
温湿度传感器 (应用配置，可能需要重启)
  ↓ 返回响应
serial-collector (封装响应为二进制帧)
  ↓ (Netty TCP，二进制帧)
rtu-gateway (解析响应，判断配置是否成功)
```

#### 认证流程（RTU 首次连接）
```
1. 管理员在 web-server 前端注册 RTU-001
   → web-server 生成 secret（如：abc123xyz）
   → 存入数据库 rtu_gateway 表

2. 边缘设备配置文件写入：
   rtuId=RTU-001
   secret=abc123xyz
   gatewayHost=192.168.1.100
   gatewayPort=8888
   modbusDeviceAddress=0x01

3. serial-collector 启动，连接到 rtu-gateway
   → 构建认证帧 (帧类型0x01)
   [AA 55 01 00 30 {rtuId:16字节} {SHA256(secret):32字节} XX]
   → 发送二进制认证帧

4. rtu-gateway 收到认证帧
   → 解析帧头 0xAA55，识别帧类型 0x01
   → 提取 rtuId 和 secret哈希
   → 调用 web-server API：POST /api/rtu/gateway/verify
     请求体：{"rtuId": "RTU-001", "secretHash": "..."}
   → web-server 查询数据库验证
   → 返回：{"valid": true, "status": "ENABLED"}

5. rtu-gateway 返回认证响应
   → 构建认证响应帧 (帧类型0x02)
   [AA 55 02 00 01 01 XX]  // 01=成功, 00=失败
   → 建立 channel 映射：channelMap.put("RTU-001", channel)

6. serial-collector 收到认证成功
   → 启动心跳任务（每30秒）
   → 启动数据采集任务（每1秒）
```

6. rtu-gateway 验证通过
   → 建立映射：channelMap.put("RTU-001", channel)
   → 建立映射：reverseMap.put(channel, "RTU-001")
   → 为 RTU-001 创建 MQTT 客户端（username=RTU-001, password=abc123xyz）→ 订阅配置主题：iot/rtu/RTU-001/device/+/config
   → 构建认证响应帧 (帧类型0x02)
   [AA 55 02 00 01 01 XX]
   → 返回认证成功

7. 后续数据传输
   → serial-collector 轮询各设备，发送数据帧 (帧类型0x03)
   → rtu-gateway 根据 channel 知道是 RTU-001
   → rtu-gateway 从 Modbus 报文提取设备地址，组合 deviceId
   → 发布到 MQTT：iot/rtu/RTU-001/device/{modbusAddress}/data
```

## 3. 模块设计

### 3.1 serial-collector（串口采集模块）

#### 职责
- 通过 Modbus-RTU 协议读取温湿度传感器数据
- 每秒采集一次数据（功能码 0x03 读取寄存器）
- **使用自定义二进制协议**封装 Modbus 数据并发送到 rtu-gateway
- 作为 Netty TCP 客户端连接到 rtu-gateway
- **完全不涉及 JSON**，所有通信使用二进制帧格式
- 断网时本地缓存数据，重连后补发
- 接收配置下发（Modbus 命令）并转发到串口设备

#### 核心功能
1. **Modbus-RTU 串口通信**
   - 串口参数：8位数据位、无校验、1位停止位、CRC16校验
   - 支持波特率：1200/2400/4800/9600/19200/38400/57600/115200（默认4800）
   - 实现 Modbus-RTU 主机功能码：
     - 0x03：读保持寄存器（读取温湿度数据）
     - 0x06：写单个寄存器（修改设备地址、波特率）

2. **定时采集（Modbus 0x03 读取）**
   - 使用 ScheduledExecutorService 实现 1 秒定时任务
   - 读取寄存器 0x0000-0x0001（湿度+温度）
   - 请求帧示例：`01 03 00 00 00 02 C4 0B`
   - 收到响应后封装为自定义二进制帧发送
   - 采集间隔可通过配置动态调整

3. **自定义二进制协议通信**
   - 所有 TCP 通信使用自定义二进制帧格式
   - 帧结构：帧头(2字节) + 帧类型(1字节) + 数据长度(2字节) + 数据体(N字节) + 校验和(1字节)
   - 帧头固定为 0xAA55，用于帧同步
   - 支持的帧类型：
     - 0x01：认证请求（rtuId + secret）
     - 0x02：认证响应
     - 0x03：Modbus 数据帧
     - 0x04：Modbus 命令帧
     - 0x05：心跳请求
     - 0x06：心跳响应

4. **Netty TCP 客户端**
   - 连接到 rtu-gateway 的 8888 端口
   - 连接成功后发送认证帧（二进制格式，包含 rtuId + secret）
   - 认证成功后，数据传输使用二进制帧封装 Modbus 报文
   - 使用心跳机制保持连接活跃（每30秒发送心跳帧）

5. **本地缓存机制**
   - 断网时数据写入本地文件（按日期分文件）
   - 内存队列缓存最近 10000 条二进制帧
   - 重连成功后按时间顺序补发历史数据
   - 补发完成后删除缓存文件

6. **配置下发执行**
   - 接收 rtu-gateway 转发的 **Modbus 写入命令**（封装在二进制帧中）
   - 解析二进制帧，提取 Modbus 命令
   - 直接转发到串口，无需理解命令含义
   - 等待设备响应并封装为二进制帧返回给 rtu-gateway
   - 支持的配置操作：
     - 修改设备地址（寄存器 0x07D0）
     - 修改波特率（寄存器 0x07D1）
     - 修改温度校准值（寄存器 0x0050）
     - 修改湿度校准值（寄存器 0x0051）

#### 技术选型
- Java 11+ 或 C/C++（适合嵌入式设备）
- Netty 4.x（TCP 客户端）
- jSerialComm 或 jlibmodbus（串口通信 + Modbus-RTU 协议）
- 无需 JSON 库

### 3.2 rtu-gateway（RTU 网关模块）

#### 职责
- 作为 Netty TCP 服务端接收多个 serial-collector 连接
- **解析自定义二进制帧协议**，识别帧类型和数据
- 通过二进制认证帧识别和认证 rtuId
- **通过 channel 映射识别数据来源**（认证时建立 channel ↔ rtuId 映射）
- **解析 Modbus-RTU 原始报文**，提取温湿度数据
- 为每个 RTU 创建独立的 MQTT 客户端
- 转发数据到 MQTT Broker（JSON 格式）
- 接收 MQTT 配置（JSON），转换为 Modbus 命令，封装为二进制帧下发

#### 核心功能
1. **Netty TCP 服务端**
   - 监听 8888 端口
   - 支持多个客户端同时连接
   - 每个连接对应一个 ChannelHandler
   - 使用自定义二进制帧解码器（BinaryFrameDecoder）

2. **二进制帧解析**
   - 识别帧头 0xAA55，实现帧同步
   - 解析帧类型、数据长度、数据体
   - 验证校验和（异或校验）
   - 支持的帧类型：
     - 0x01：认证请求
     - 0x03：Modbus 数据
     - 0x05：心跳请求

3. **认证机制**
   - 接收认证帧（0x01），提取 rtuId（16字节）和 secret（32字节）
   - 调用 web-server API 验证：
     ```
     GET /api/rtu/gateway/{rtuId}/verify?secret={secret}
     Response: {"valid": true, "status": "ENABLED"}
     ```
   - 认证成功后：
     - 建立双向映射：
       - `channelMap.put("RTU-001", channel)` - rtuId → channel
       - `reverseMap.put(channel, "RTU-001")` - channel → rtuId
     - 为该 rtuId 创建独立的 MQTT 客户端
     - 订阅配置主题：`iot/rtu/RTU-001/config`
     - 返回认证响应帧（0x02）
   - 认证失败则关闭连接并记录日志

4. **连接管理**
   - 维护双向映射：
     - `Map<String, Channel> channelMap` - 用于配置下发时查找 channel
     - `Map<Channel, String> reverseMap` - 用于数据上报时识别 rtuId
   - 连接断开时清理映射关系和 MQTT 客户端
   - 支持同一 rtuId 的重连（踢掉旧连接）
   - 认证超时机制：连接建立后 30 秒内未认证则关闭

5. **Modbus 协议解析**
   - 接收 Modbus 数据帧（0x03），提取 Modbus-RTU 原始报文
   - 通过 channel 从 reverseMap 获取 rtuId
   - 解析 Modbus 响应帧：
     - 验证 CRC16 校验
     - 提取湿度：寄存器值 / 10.0
     - 提取温度：寄存器值 / 10.0（负数处理补码）
   - 支持功能码 0x03（读保持寄存器）

6. **MQTT 客户端管理**
   - 为每个 rtuId 创建独立的 MQTT 客户端
   - MQTT 连接参数：
     - Broker: `tcp://localhost:1883`
     - ClientId: `rtu-gateway-{rtuId}`
     - Username: `{rtuId}`
     - Password: `{secret}`
   - 发布主题：
     - 数据: `iot/rtu/{rtuId}/data`
     - 心跳: `iot/rtu/{rtuId}/heartbeat`（每 30 秒）
   - 订阅主题：
     - 配置: `iot/rtu/{rtuId}/config`

7. **消息转发**
   - 接收到 Modbus 原始报文后解析并封装为 JSON 格式发布到 MQTT
   ```json
   {
     "rtuId": "RTU-001",
     "temp": 26.5,
     "humidity": 58.4,
     "timestamp": 1741587654000
   }
   ```

8. **配置下发**
   - 订阅 `iot/rtu/{rtuId}/config` 主题
   - 接收到 JSON 配置后，**转换为 Modbus 写入命令**
   - 封装为二进制帧（0x04）并通过 channelMap 查找对应的 channel 发送
   - 配置转换示例：
     ```java

     {"baudRate": 9600}
       → Modbus: [01 06 07 D1 00 02 88 3A]
       → 二进制帧: [AA 55 04 00 08 01 06 07 D1 00 02 88 3A XX]

   - 接收 serial-collector 返回的设备响应（二进制帧 0x03），判断配置是否成功

   - 接收心跳帧（0x05）

   - 可选：返回心跳响应帧（0x06）

### 4.1 serial-collector ↔ 温湿度传感器（Modbus-RTU）

#### 串口参数
- 编码：8位二进制
- 数据位：8位
- 奇偶校验：无
- 停止位：1位
- 错误校验：CRC16（Modbus标准）
- 波特率：1200/2400/4800/9600/19200/38400/57600/115200（默认4800）

#### Modbus-RTU 帧格式
```
初始结构：≥4字节时间间隔
地址码：1字节
功能码：1字节
数据区：N字节
CRC校验：2字节（低字节在前，高字节在后）
结束结构：≥4字节时间间隔
```

#### 寄存器地址映射
| 寄存器地址 | 功能 | 支持功能码 | 说明 |
|-----------|------|-----------|------|
| 0x0000 | 湿度 | 0x03/0x04 | 湿度实时值（扩大10倍） |
| 0x0001 | 温度 | 0x03/0x04 | 温度实时值（扩大10倍，负数为补码） |
| 0x0050 | 温度校准值 | 0x03/0x04/0x06 | 整数（扩大10倍） |
| 0x0051 | 湿度校准值 | 0x03/0x04/0x06 | 整数（扩大10倍） |
| 0x07D0 | 设备地址 | 0x03/0x04/0x06 | 1-254（默认1） |
| 0x07D1 | 波特率 | 0x03/0x04/0x06 | 0=2400, 1=4800, 2=9600, 3=19200, 4=38400, 5=57600, 6=115200, 7=1200 |

#### 读取温湿度（功能码 0x03）
**请求帧示例**（读取设备地址0x01的温湿度）：
```
地址码  功能码  起始地址    数据长度    CRC低   CRC高
0x01    0x03    0x00 0x00   0x00 0x02   0xC4    0x0B
```

**响应帧示例**（温度-10.1℃，湿度65.8%RH）：
```
地址码  功能码  字节数  湿度值      温度值      CRC低   CRC高
0x01    0x03    0x04    0x02 0x92   0xFF 0x9B   0x5A    0x3D
```

**数据解析**：
- 湿度：0x0292 = 658 → 658 / 10.0 = 65.8%RH
- 温度：0xFF9B（补码）= -101 → -101 / 10.0 = -10.1℃

#### 修改设备配置（功能码 0x06）
**请求帧示例**（修改波特率为9600，代码=2）：
```
地址码  功能码  寄存器地址  修改数值    CRC低   CRC高
0x01    0x06    0x07 0xD1   0x00 0x02   0x88    0x3A
```

**响应帧**（原样返回表示成功）：
```
0x01    0x06    0x07 0xD1   0x00 0x02   0x88    0x3A
```

**注意**：修改设备地址或波特率后需断电重启设备。

### 4.2 serial-collector ↔ rtu-gateway（自定义二进制协议）

#### 协议说明

**核心原则**：serial-collector ↔ rtu-gateway 之间**完全使用二进制协议，不涉及 JSON**

- ✅ 认证：自定义二进制帧（rtuId + secret）
- ✅ 数据上报：二进制帧封装 Modbus 原始报文
- ✅ 配置下发：二进制帧封装 Modbus 写入命令
- ✅ 心跳：自定义二进制帧

#### 帧格式定义

```
+--------+--------+----------+----------+--------+
| 帧头   | 帧类型 | 数据长度 | 数据体   | 校验和 |
| 2字节  | 1字节  | 2字节    | N字节    | 1字节  |
+--------+--------+----------+----------+--------+

帧头：固定为 0xAA55（用于帧同步）
帧类型：
  0x01 = 认证请求
  0x02 = 认证响应
  0x03 = Modbus 数据（上报/响应）
  0x04 = Modbus 命令（下发）
  0x05 = 心跳请求
  0x06 = 心跳响应
数据长度：数据体的字节数（大端序，2字节）
数据体：具体内容
校验和：所有字节（除校验和本身）的异或值
```

#### 认证帧（0x01）

**collector → gateway：**

```
帧头: AA 55
帧类型: 01
数据长度: 00 30 (48字节)
数据体:
  rtuId: 16字节（UTF-8明文，不足补0）
    例如："RTU-001\0\0\0\0\0\0\0\0\0"
  secret: 32字节（SHA-256哈希值）
    计算方式：SHA256(原始secret + rtuId + timestamp前4字节)
校验和: XX

完整示例：
AA 55 01 00 30
52 54 55 2D 30 30 31 00 00 00 00 00 00 00 00 00  // "RTU-001" (明文)
A3 B2 C1 D4 E5 F6 07 18 29 3A 4B 5C 6D 7E 8F 90  // SHA-256哈希值（32字节）
A1 B2 C3 D4 E5 F6 F7 F8 F9 FA FB FC FD FE FF 00
XX
```

**加密说明**：
- rtuId 使用明文传输（用于识别身份）
- secret 使用 SHA-256 单向哈希加密
- 哈希输入：`原始secret + rtuId + 当前时间戳前4字节`
- 防重放：时间戳参与哈希计算，gateway 验证时间戳有效期（±5分钟）
- 防篡改：整个帧有校验和保护

**gateway → collector：**

```
帧头: AA 55
帧类型: 02
数据长度: 00 01
数据体:
  结果: 1字节
    0x00 = 认证失败
    0x01 = 认证成功
校验和: XX

成功示例：
AA 55 02 00 01 01 XX

失败示例：
AA 55 02 00 01 00 XX
```

#### Modbus 数据帧（0x03）

**collector → gateway（数据上报）：**

```
帧头: AA 55
帧类型: 03
数据长度: 00 09 (9字节)
数据体: Modbus 响应帧（原始字节流）
  01 03 04 02 92 FF 9B 5A 3D
校验和: XX

完整示例：
AA 55 03 00 09
01 03 04 02 92 FF 9B 5A 3D
XX
```

**说明**：
- 数据体就是 Modbus-RTU 的完整响应帧
- gateway 从 Modbus 帧第一个字节提取设备地址（0x01，通常固定）
- gateway 通过 channel 映射识别 rtuId（RTU-001）
- 由于一对一架构，rtuId 即代表该设备

#### Modbus 命令帧（0x04）

**gateway → collector（配置下发）：**

```
帧头: AA 55
帧类型: 04
数据长度: 00 08 (8字节)
数据体: Modbus 写入命令（原始字节流）
  01 06 07 D1 00 02 88 3A
校验和: XX

完整示例：
AA 55 04 00 08
01 06 07 D1 00 02 88 3A
XX
```

**collector → gateway（命令响应）：**

```
帧头: AA 55
帧类型: 03 (复用数据帧类型)
数据长度: 00 08
数据体: Modbus 设备响应
  01 06 07 D1 00 02 88 3A
校验和: XX
```

#### 心跳帧（0x05/0x06）

**collector → gateway（心跳请求）：**

```
帧头: AA 55
帧类型: 05
数据长度: 00 04
数据体: 时间戳（4字节，Unix时间戳，大端序）
校验和: XX

示例：
AA 55 05 00 04
67 8A BC DE  // 时间戳
XX
```

**gateway → collector（心跳响应，可选）：**

```
帧头: AA 55
帧类型: 06
数据长度: 00 01
数据体:
  0x01 = 心跳确认
校验和: XX
```

### 4.2 serial-collector ↔ rtu-gateway（Netty TCP）

#### 协议说明

**重要**：该协议采用**混合格式**：
- **认证阶段**：使用 JSON 格式（以换行符 `\n` 结尾）
- **数据传输阶段**：使用 Modbus-RTU 原始字节流（无 JSON 包装）
- **配置下发**：使用 JSON 格式（以换行符 `\n` 结尾）

#### rtuId 识别机制

rtu-gateway 通过 **channel 映射**识别数据来源：

```
认证阶段（建立映射）：
  serial-collector 发送 JSON: {"type":"AUTH", "rtuId":"RTU-001", ...}
  ↓
  rtu-gateway 验证成功后建立映射：
    channelMap.put("RTU-001", channel)
    reverseMap.put(channel, "RTU-001")

数据传输阶段（使用映射）：
  serial-collector 发送原始字节流: [01 03 04 02 92 FF 9B 5A 3D]
  ↓
  rtu-gateway 从 channel 识别 rtuId:
    rtuId = reverseMap.get(channel)  // 得到 "RTU-001"
```

#### 认证消息（AUTH）- JSON 格式

```json
{
  "type": "AUTH",
  "rtuId": "RTU-001",
  "secret": "abc123xyz",
  "timestamp": 1741587654000
}\n
```

**说明**：
- 连接建立后 30 秒内必须发送
- 以换行符 `\n` 结尾
- rtu-gateway 验证后建立 channel ↔ rtuId 映射
 认证响应（AUTH_RESPONSE）- JSON 格式

```json
{
  "type": "AUTH_RESPONSE",
  "success": true,
  "message": "Authentication successful"
}\n
```

#### 数据消息（DATA）- Modbus 原始字节流

```
十六进制字节流（无 JSON 包装）：
01 03 04 02 92 FF 9B 5A 3D

解析：
  地址码: 01
  功能码: 03 (读保持寄存器)
  字节数: 04
  湿度: 02 92 (十六进制) = 658 → 65.8%RH
  温度: FF 9B (十六进制，补码) = -101 → -10.1℃
  CRC: 5A 3D
```

**说明**：
- serial-collector 直接转发 Modbus 响应帧
- rtu-gateway 通过 channel 识别 rtuId
- rtu-gateway 负责解析 Modbus 协议

#### 心跳消息（HEARTBEAT）- JSON 格式

```json
{
  "type": "HEARTBEAT",
  "timestamp": 1741587654000
}\n
```

**说明**：
- serial-collector 每 30 秒发送一次
- 用于保持 TCP 连接活跃

#### 配置消息（CONFIG）- JSON 格式

```json
{
  "type": "CONFIG",
  "interval": 5,
  "modbusDeviceAddress": 1,
  "baudRate": 4800,
  "tempCalibration": 0.0,
  "humidityCalibration": 0.0,
  "timestamp": 1741587654000
}\n
```

**说明**：
- rtu-gateway 从 MQTT 接收配置后转发给 serial-collector
- serial-collector 执行 Modbus 写入操作

#### 配置响应（CONFIG_RESPONSE）- JSON 格式

```json
{
  "type": "CONFIG_RESPONSE",
  "success": true,
  "needRestart": false,
  "message": "Configuration applied successfully",
  "timestamp": 1741587654000
}\n
```

**说明**：
- `needRestart`: 如果修改了设备地址或波特率，需要断电重启设备

## 8. 关键技术点

### 8.1 Modbus-RTU 协议实现

#### CRC16 校验算法
```java
public static byte[] calculateCRC16(byte[] data) {
    int crc = 0xFFFF;
    for (byte b : data) {
        crc ^= (b & 0xFF);
        for (int i = 0; i < 8; i++) {
            if ((crc & 0x0001) != 0) {
                crc >>= 1;
                crc ^= 0xA001;
            } else {
                crc >>= 1;
            }
        }
    }
    return new byte[]{(byte) (crc & 0xFF), (byte) ((crc >> 8) & 0xFF)};
}
```

#### 温度负数处理（补码转换）
```java
public static double parseTemperature(int rawValue) {
    // 如果最高位为1，表示负数（补码）
    if ((rawValue & 0x8000) != 0) {
        // 补码转原码：取反加1
        rawValue = -(~rawValue & 0xFFFF) - 1;
    }
    return rawValue / 10.0;
}
```

#### Modbus 读取温湿度示例
```java
public TempHumidityData readTempHumidity(int deviceAddress) throws IOException {
    // 构造请求帧：读取寄存器 0x0000-0x0001
    byte[] request = new byte[]{
        (byte) deviceAddress,  // 设备地址
        0x03,                  // 功能码：读保持寄存器
        0x00, 0x00,           // 起始地址：0x0000
        0x00, 0x02            // 寄存器数量：2
    };

    // 计算并添加 CRC
    byte[] crc = calculateCRC16(request);
    byte[] frame = new byte[request.length + 2];
    System.arraycopy(request, 0, frame, 0, request.length);
    frame[request.length] = crc[0];
    frame[request.length + 1] = crc[1];

    // 发送请求
    serialPort.writeBytes(frame, frame.length);

    // 等待响应（至少9字节：地址+功能码+字节数+4字节数据+2字节CRC）
    Thread.sleep(100);
    byte[] response = serialPort.readBytes(9);

    // 验证 CRC
    byte[] dataPart = Arrays.copyOfRange(response, 0, 7);
    byte[] receivedCRC = Arrays.copyOfRange(response, 7, 9);
    byte[] calculatedCRC = calculateCRC16(dataPart);
    if (!Arrays.equals(receivedCRC, calculatedCRC)) {
        throw new IOException("CRC check failed");
    }

    // 解析数据
    int humidityRaw = ((response[3] & 0xFF) << 8) | (response[4] & 0xFF);
    int tempRaw = ((response[5] & 0xFF) << 8) | (response[6] & 0xFF);

    double humidity = humidityRaw / 10.0;
    double temperature = parseTemperature(tempRaw);

    return new TempHumidityData(temperature, humidity);
}
```

#### Modbus 写入寄存器示例
```java
public boolean writeRegister(int deviceAddress, int registerAddress, int value) throws IOException {
    // 构造请求帧：写单个寄存器
    byte[] request = new byte[]{
        (byte) deviceAddress,
        0x06,  // 功能码：写单个寄存器
        (byte) ((registerAddress >> 8) & 0xFF),
        (byte) (registerAddress & 0xFF),
        (byte) ((value >> 8) & 0xFF),
        (byte) (value & 0xFF)
    };

    // 计算并添加 CRC
    byte[] crc = calculateCRC16(request);
    byte[] frame = new byte[request.length + 2];
    System.arraycopy(request, 0, frame, 0, request.length);
    frame[request.length] = crc[0];
    frame[request.length + 1] = crc[1];

    // 发送请求
    serialPort.writeBytes(frame, frame.length);

    // 等待响应（写入成功时原样返回）
    Thread.sleep(100);
    byte[] response = serialPort.readBytes(8);

    // 验证响应是否与请求一致
    return Arrays.equals(frame, response);
}
```

#### 配置下发执行流程
```java
public void applyConfig(ConfigMessage config) throws IOException {
    // 1. 修改波特率（如果需要）
    if (config.getBaudRate() != currentBaudRate) {
        writeRegister(currentDeviceAddress, 0x07D1, getBaudRateCode(config.getBaudRate()));
        needRestart = true;
    }

    // 2. 修改设备地址（如果需要）
    if (config.getModbusDeviceAddress() != currentDeviceAddress) {
        writeRegister(currentDeviceAddress, 0x07D0, config.getModbusDeviceAddress());
        needRestart = true;
    }

    // 3. 写入温度校准值
    if (config.getTempCalibration() != null) {
        int calibValue = (int) (config.getTempCalibration() * 10);
        writeRegister(config.getModbusDeviceAddress(), 0x0050, calibValue);
    }

    // 4. 写入湿度校准值
    if (config.getHumidityCalibration() != null) {
        int calibValue = (int) (config.getHumidityCalibration() * 10);
        writeRegister(config.getModbusDeviceAddress(), 0x0051, calibValue);
    }

    // 5. 更新采集间隔（软件层面）
    if (config.getInterval() != null) {
        updateCollectionInterval(config.getInterval());
    }

    // 6. 返回响应
    if (needRestart) {
        return ConfigResponse.nert("Device address or baud rate changed");
    } else {
        return ConfigResponse.success();
    }
}

private int getBaudRateCode(int baudRate) {
    switch (baudRate) {
        case 1200: return 7;
        case 2400: return 0;
        case 4800: return 1;
        case 9600: return 2;
        case 19200: return 3;
        case 38400: return 4;
        case 57600: return 5;
        case 115200: return 6;
        default: throw new IllegalArgumentException("Unsupported baud rate: " + baudRate);
    }
}
```

### 8.2 连接管理与重连机制

#### serial-collector 重连策略
- 初始重连间隔：5 秒
- 最大重连间隔：60 秒
- 指数退避算法：每次失败后间隔翻倍
- 无限重试直到连接成功

#### rtu-gateway 连接管理
- 维护 rtuId 到 Channel 的映射
- 检测到连接断开时清理映射
- 支持同一 rtuId 的重连（踢掉旧连接）

### 8.2 数据缓存与补发

#### 缓存策略
- 内存队列：LinkedBlockingQueue，容量 10000
- 文件缓存：按日期分文件，JSON Lines 格式
- 缓存目录：`./data/cache/{rtuId}/`

#### 补发策略
- 重连成功后先发送内存队列数据
- 再按时间顺序读取文件缓存并发送
- 每批发送 100 条，间隔 100ms
- 补发完成后删除缓存文件

### 8.3 消息过滤与验证

#### web-server 消息过滤流程
```java
public void onMessage(String topic, MqttMessage message) {
    String rtuId = extractRtuIdFromTopic(topic);

    // 查询网关信息
    RtuGateway gateway = rtuGatewayRepository.findByRtuId(rtuId);

    // 过滤逻辑
    if (gateway == null) {
        log.warn("Unknown RTU: {}, message dropped", rtuId);
        return;
    }

    if ("DISABLED".equals(gateway.getStatus())) {
        log.warn("Disabled RTU: {}, message dropped", rtuId);
        return;
    }

    // 处理消息
    processMessage(gateway, message);
}
```

### 核心设计要点

#### 1. 纯字节流通信
- serial-collector ↔ rtu-gateway 使用自定义二进制帧协议
- 帧头 0xAA55，支持帧同步和校验
- 无 JSON 依赖，适合嵌入式设备
- 可用 C/C++/Java 等任何语言实现

#### 2. 一对一架构
- 串口是点对点通信，一个 RTU 对应一个串口，一个串口对应一个 Modbus 设备
- 设备标识：rtuId 即代表该设备
- 数据库层面：rtu_gateway（RTU网关）直接管理数据
- Modbus 设备地址通常固定为 0x01

#### 3. 协议转换
- serial-collector：Modbus-RTU ↔ 二进制帧
- rtu-gateway：二进制帧 ↔ JSON（MQTT）
- web-server：JSON 处理和存储

#### 4. 数据流标识
```
上行：设备 → collector → gateway（从Modbus报文提取地址） → MQTT（deviceId） → web-server
下行：web-server → MQTT（deviceId） → gateway（转换为Modbus命令） → collector → 设备
```

#### 5. 关键表结构
- `rtu_gateway`：RTU 网关（边缘设备）
- `rtu_device`：设备（传感器），外键关联 rtu_gateway
- `rtu_data`：采集数据，包含 device_id 和 rtu_id
- `rtu_config`：设备配置，包含 Modbus 参数和报警阈值
- `rtu_alarm`：报警记录
