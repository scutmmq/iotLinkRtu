## Modbus-RTU 协议

### 概述

Modbus-RTU 是一种串行通信协议，用于 serial-collector 与温湿度传感器设备之间的通信。

### 支持的功能码

| 功能码 | 名称 | 用途 |
|--------|------|------|
| 0x03 | 读保持寄存器 | 读取温湿度数据 |
| 0x06 | 写单个寄存器 | 修改设备配置（地址、波特率、校准值） |

### 寄存器地址映射

| 寄存器地址 | 数据类型 | 说明 | 单位 |
|-----------|---------|------|------|
| 0x0000 | 16-bit 无符号整数 | 湿度值（扩大10倍） | 0.1%RH |
| 0x0001 | 16-bit 有符号整数（补码） | 温度值（扩大10倍） | 0.1℃ |
| 0x0050 | 16-bit 有符号整数 | 温度校准值 | 0.1℃ |
| 0x0051 | 16-bit 有符号整数 | 湿度校准值 | 0.1%RH |
| 0x07D0 | 16-bit 无符号整数 | 设备地址 | 1-254 |
| 0x07D1 | 16-bit 无符号整数 | 波特率索引 | 见下表 |

#### 波特率索引表

| 索引值 | 波特率 |
|--------|--------|
| 0 | 1200 |
| 1 | 2400 |
| 2 | 4800 |
| 3 | 9600 |
| 4 | 19200 |
| 5 | 38400 |
| 6 | 57600 |
| 7 | 115200 |

### 功能码 0x03：读保持寄存器

#### 请求帧格式

| 字段 | 字节数 | 说明 | 示例值 |
|------|--------|------|--------|
| 设备地址 | 1 | 从站地址（0x01-0xFE） | 0x01 |
| 功能码 | 1 | 固定为 0x03 | 0x03 |
| 起始地址高字节 | 1 | 寄存器起始地址高8位 | 0x00 |
| 起始地址低字节 | 1 | 寄存器起始地址低8位 | 0x00 |
| 寄存器数量高字节 | 1 | 要读取的寄存器数量高8位 | 0x00 |
| 寄存器数量低字节 | 1 | 要读取的寄存器数量低8位 | 0x02 |
| CRC 低字节 | 1 | CRC16 校验低字节 | 0xC4 |
| CRC 高字节 | 1 | CRC16 校验高字节 | 0x0B |

**示例：读取温湿度（设备地址 0x01）**

```
请求: 01 03 00 00 00 02 C4 0B
```

#### 响应帧格式

| 字段 | 字节数 | 说明 | 示例值 |
|------|--------|------|--------|
| 设备地址 | 1 | 从站地址 | 0x01 |
| 功能码 | 1 | 固定为 0x03 | 0x03 |
| 字节数 | 1 | 后续数据字节数 | 0x04 |
| 湿度高字节 | 1 | 湿度值高8位 | 0x02 |
| 湿度低字节 | 1 | 湿度值低8位 | 0x92 |
| 温度高字节 | 1 | 温度值高8位 | 0xFF |
| 温度低字节 | 1 | 温度值低8位 | 0x9B |
| CRC 低字节 | 1 | CRC16 校验低字节 | 0x5A |
| CRC 高字节 | 1 | CRC16 校验高字节 | 0x3D |

**示例：温度 -10.1℃，湿度 65.8%RH**

```
响应: 01 03 04 02 92 FF 9B 5A 3D

解析:
- 湿度原始值 = 0x0292 = 658 → 65.8%RH
- 温度原始值 = 0xFF9B = -101（补码） → -10.1℃
```

#### 温度补码处理

温度使用 16-bit 有符号整数（补码）表示：

- **正温度**：0x0000 ~ 0x7FFF（0 ~ 32767）
- **负温度**：0x8000 ~ 0xFFFF（-32768 ~ -1）

**转换公式：**

```java
int tempRaw = (highByte << 8) | lowByte;
if (tempRaw > 32767) {
    tempRaw = tempRaw - 65536;  // 补码转换
}
double temperature = tempRaw / 10.0;
```

**示例：**

| 原始值 | 十进制 | 实际温度 |
|--------|--------|----------|
| 0x0000 | 0 | 0.0℃ |
| 0x0064 | 100 | 10.0℃ |
| 0x03E8 | 1000 | 100.0℃ |
| 0xFFFF | -1 | -0.1℃ |
| 0xFF9B | -101 | -10.1℃ |
| 0xFE70 | -400 | -40.0℃ |

### 功能码 0x06：写单个寄存器

#### 请求帧格式

| 字段 | 字节数 | 说明 | 示例值 |
|------|--------|------|--------|
| 设备地址 | 1 | 从站地址 | 0x01 |
| 功能码 | 1 | 固定为 0x06 | 0x06 |
| 寄存器地址高字节 | 1 | 寄存器地址高8位 | 0x07 |
| 寄存器地址低字节 | 1 | 寄存器地址低8位 | 0xD0 |
| 寄存器值高字节 | 1 | 要写入的值高8位 | 0x00 |
| 寄存器值低字节 | 1 | 要写入的值低8位 | 0x05 |
| CRC 低字节 | 1 | CRC16 校验低字节 | - |
| CRC 高字节 | 1 | CRC16 校验高字节 | - |

**示例：修改设备地址为 0x05**

```
请求: 01 06 07 D0 00 05 [CRC_L] [CRC_H]
```

#### 响应帧格式

响应帧与请求帧完全相同（回显），表示写入成功。

### CRC16 校验算法

Modbus-RTU 使用 CRC16-MODBUS 算法（多项式 0xA001）。

**算法步骤：**

1. 初始化 CRC 寄存器为 0xFFFF
2. 对每个数据字节：
   - CRC 与数据字节异或
   - 对 8 个比特位循环：
     - 如果 CRC 最低位为 1，右移 1 位后与 0xA001 异或
     - 否则，仅右移 1 位
3. 返回 CRC 低字节在前，高字节在后

**Java 实现：**

```java
public static byte[] calculateCRC16(byte[] data, int offset, int length) {
    int crc = 0xFFFF;

    for (int i = offset; i < offset + length; i++) {
        crc ^= (data[i] & 0xFF);

        for (int j = 0; j < 8; j++) {
            if ((crc & 0x0001) != 0) {
                crc >>= 1;
                crc ^= 0xA001;
            } else {
                crc >>= 1;
            }
        }
    }

    return new byte[]{
        (byte) (crc & 0xFF),        // 低字节
        (byte) ((crc >> 8) & 0xFF)  // 高字节
    };
}
```

## 自定义二进制协议

### 概述

自定义二进制协议用于 serial-collector 与 rtu-gateway 之间的 TCP 通信，封装 Modbus 数据、认证信息和心跳消息。

### 帧结构

所有帧遵循统一的格式：

| 字段 | 字节数 | 说明 |
|------|--------|------|
| 帧头 | 2 | 固定为 0xAA55（帧同步标识） |
| 帧类型 | 1 | 标识帧的用途（见下表） |
| 数据长度 | 2 | 数据体长度（大端序，不包含帧头、类
| 数据体 | N | 具体数据内容（长度由"数据长度"字段指定） |
| 校验和 | 1 | 异或校验（从帧头到数据体末尾的所有字节） |

**总帧长 = 2 + 1 + 2 + N + 1 = N + 6 字节**

### 帧类型定义

| 类型值 | 名称 | 方向 | 说明 |
|--------|------|------|------|
| 0x01 | 认证请求 | Collector → Gateway | RTU 连接后发送认证信息 |
| 0x02 | 认证响应 | Gateway → Collector | 网关返回认证结果 |
| 0x03 | Modbus 数据 | Collector → Gateway | 上报温湿度数据 |
| 0x04 | Modbus 命令 | Gateway → Collector | 下发配置命令 |
| 0x05 | 心跳请求 | Collector → Gateway | 保持连接活跃 |
| 0x06 | 心跳响应 | Gateway → Collector | 确认心跳 |

### 帧类型详解

#### 1. 认证请求帧（0x01）

**用途：** serial-collector 连接到 rtu-gateway 后，立即发送认证请求。

**数据体结构：**

| 字段 | 字节数 | 说明 |
|------|--------|------|
| rtuId | 16 | RTU 标识符（UTF-8 明文，不足补 0x00） |
| secretHash | 32 | SHA-256 哈希值（见下方算法） |

**总长度：** 2 + 1 + 2 + 48 + 1 = 54 字节

**SHA-256 哈希算法：**

```
输入: secret（明文） + rtuId（明文） + timestamp（4字节，当前Unix时间戳）
输出: 32字节哈希值
```

**Java 实现：**

```java
MessageDigest digest = MessageDigest.getInstance("SHA-256");
digest.update(secret.getBytes(StandardCharsets.UTF_8));
digest.update(rtuId.getBytes(StandardCharsets.UTF_8));
digest.update(timestampBytes);  // 4字节时间戳
byte[] hash = digest.digest();  // 32字节
```

**示例：**

```
rtuId = "ZM611116"
secret = "momingqinxlinknb"
timestamp = 0x65A3B2C1 (1705234113)

帧结构:
AA 55 01 00 30 [rtuId 16字节] [SHA-256 32字节] [校验和]
```

**安全性：**

- **防明文传输：** secret 不直接传输，仅传输哈希值
- **防重放攻击：** 哈希包含时间戳，每次连接哈希值不同
- **防篡改：** 网关验证 SHA-256(secret + rtuId + timestamp) 是否匹配

#### 2. 认证响应帧（0x02）

**用途：** rtu-gateway 返回认证结果。

**数据体结构：**

| 字段 | 字节数 | 说明 |
|------|--------|------|
| result | 1 | 0x01=成功，0x00=失败 |

**总长度：** 2 + 1 + 2 + 1 + 1 = 7 字节

**示例：**

```
认证成功: AA 55 02 00 01 01 [校验和]
认证失败: AA 55 02 00 01 00 [校验和]
```

**行为：**

- **成功：** serial-collector 启动心跳任务和数据补发
- **失败：** serial-collector 关闭连接

#### 3. Modbus 数据帧（0x03）

**用途：** serial-collector 将从串口读取的 Modbus 响应封装后上报给 rtu-gateway。

**数据体结构：**

| 字段 | 字节数 | 说明 |
|------|--------|------|
| modbusData | N | 完整的 Modbus-RTU 响应帧（包含 CRC） |

**总长度：** 2 + 1 + 2 + N + 1 = N + 6 字节

**示例：**

```
Modbus 响应: 01 03 04 02 92 FF 9B 5A 3D (9字节)

封装后的二进制帧:
AA 55 03 00 09 01 03 04 02 92 FF 9B 5A 3D [校验和]

字段解析:
- AA 55: 帧头
- 03: 帧类型（Modbus 数据）
- 00 09: 数据长度（9字节）
- 01 03 04 02 92 FF 9B 5A 3D: Modbus 原始响应
- [校验和]: 异或校验
```

**网关处理：**

1. 解析二进制帧，提取 Modbus 数据
2. 验证 Modbus CRC
3. 解析温湿度值
4. 通过 MQTT 发布到 `iot/rtu/{rtuId}/data`

#### 4. Modbus 命令帧（0x04）

**用途：** rtu-gateway 下发配置命令给 serial-collector，由 collector 转发到串口设备。

**数据体结构：**

| 字段 | 字节数 | 说明 |
|------|--------|------|
| modbusCommand | N | 完整的 Modbus-RTU 命令帧（包含 CRC） |

**总长度：** 2 + 1 + 2 + N + 1 = N + 6 字节

**示例：修改设备地址为 0x05**

```
Modbus 命令: 01 06 07 D0 00 05 [CRC_L] [CRC_H] (8字节)

封装后的二进制帧:
AA 55 04 00 08 01 06 07 D0 00 05 [CRC_L] [CRC_H] [校验和]
```

**Collector 处理流程：**

1. 接收二进制帧，提取 Modbus 命令
2. 通过串口发送命令到设备
3. 等待设备响应（超时 500ms）
4. 将设备响应封装为 Modbus 数据帧（0x03）返回网关

#### 5. 心跳请求帧（0x05）

**用途：** serial-collector 每 30 秒发送一次心跳，保持连接活跃。

**数据体结构：**

| 字段 | 字节数 | 说明 |
|------|--------|------|
| timestamp | 4 | Unix 时间戳（秒，大端序） |

**总长度：** 2 + 1 + 2 + 4 + 1 = 10 字节

**示例：**

```
AA 55 05 00 04 65 A3 B2 C1 [校验和]

解析:
- timestamp = 0x65A3B2C1 = 1705234113 (2024-01-14 12:35:13 UTC)
```

#### 6. 心跳响应帧（0x06）

**用途：** rtu-gateway 确认收到心跳。

**数据体结构：**

| 字段 | 字节数 | 说明 |
|------|--------|------|
| timesta| 回显请求中的时间戳 |

**总长度：** 2 + 1 + 2 + 4 + 1 = 10 字节

### 校验和算法

使用异或（XOR）校验，计算从帧头到数据体末尾的所有字节。

**算法：**

```java
byte checksum = 0;
for (int i = 0; i < length; i++) {
    checksum ^= data[i];
}
```

**示例：**

```
帧数据: AA 55 03 00 09 01 03 04 02 92 FF 9B 5A 3D

校验和计算:
AA ^ 55 ^ 03 ^ 00 ^ 09 ^ 01 ^ 03 ^ 04 ^ 02 ^ 92 ^ FF ^ 9B ^ 5A ^ 3D = 0xE7

完整帧: AA 55 03 00 09 01 03 04 02 92 FF 9B 5A 3D E7
```

## 数据流程

### 完整数据采集流程

```
1. serial-collector 启动
   ↓
2. 打开串口（COM3, 4800 波特率）
   ↓
3. 连接 rtu-gateway（TCP）
   ↓
4. 发送认证帧（0x01）
   ↓
5. 收到认证成功响应（0x02, result=0x01）
   ↓
6. 启动心跳任务（每 30 秒发送 0x05）
   ↓
7. 补发缓存数据（如果有）
   ↓
8. 启动数据采集任务（每 1 秒）
   ↓
9. 构建 Modbus 读取命令（01 03 00 00 00 02 C4 0B）
   ↓
10. 发送到串口
   ↓
11. 读取设备响应（01 03 04 02 92 FF 9B 5A 3D）
   ↓
12. 验证 Modbus CRC
   ↓
13. 封装为二进制帧（0x03）
   ↓
14. 发送给 rtu-gateway
   ↓
15. 如果断网，缓存到本地
   ↓
16. 重连成功后自动补发
```

### 配置下发流程

```
1. Web 前端发起配置修改请求
   ↓
2. web-server 通过 MQTT 发布到 iot/rtu/{rtuId}/config
   ↓
3. rtu-gateway 订阅到消息
   ↓
4. 构建 Modbus 命令（功能码 0x06）
   ↓
5. 封装为二进制帧（0x04）发送给 serial-collector
   ↓
6. serial-collector 提取 Modbus 命令
   ↓
7. 通过串口发送到设备
   ↓
8. 等待设备响应（超时 500ms）
   ↓
9. 封装响应为二进制帧（0x03）返回 rtu-gateway
   ↓
10. rtu-gateway 通过 MQTT 发布执行结果
   ↓
11. web-server 更新配置状态
```

### 断网重连流程

```
1. 检测到连接断开
   ↓
2. 停止心跳任务
   ↓
3. 数据采集继续进行
   ↓
4. 新数据缓存到本地（内存队列 + 文件）
   ↓
5. 每 5 秒尝试重连
   ↓
6. 连接成功后发送认证帧
   ↓
7. 认证成功后自动补发缓存数据
   ↓
8. 补发完成后清空缓存
   ↓
9. 恢复正常数据上报
```

## 错误处理

### Modbus 协议错误

| 错误类型 | 检测方法 | 处理方式 |
|---------|---------|---------|
| CRC 校验失败 | `verifyCRC()` 返回 false | 丢弃帧，记录日志，不上报 |
| 响应超时 | 500ms 内无响应 | 记录日志，跳过本次采集 |
| 功能码错误 | 响应功能码 ≠ 0x03 | 丢弃帧，记录日志 |
| 数据长度错误 | 响应长度 ≠ 9 字节 | 丢弃帧，记录日志 |
| 字节数错误 | 响应[2] ≠ 0x04 | 丢弃帧，记录日志 |

### 二进制协议错误

| 错误类型 | 检测方法 | 处理方式 |
|---------|---------|---------|
| 帧头错误 | 前2字节 ≠ 0xAA55 | 丢弃1字节，继续查找帧头 |
| 校验和错误 | 计算值 ≠ 接收值 | 丢弃帧，记录日志 |
| 数据长度异常 | 长度 > 缓冲区 | 等待更多数据或丢弃 |
| 认证失败 | 响应 result=0x00 | 关闭连接，停止服务 |

### 网络错误

| 错误类型 | 检测方法 | 处理方式 |
|---------|---------|---------|
| 连接失败 | `connect()` 异常 | 5 秒后自动重连 |
| 连接断开 | `channelInactive()` | 停止心跳，5 秒后重连 |
| 发送失败 | `writeAndFlush()` 异常 | 缓存数据到本地 |
| 心跳超时 | 90 秒无心跳响应 | 网关标记 RTU 离线 |

### 串口错误

| 错误类型 | 检测方法 | 处理方式 |
|---------|---------|---------|
| 打开失败 | `openPort()` 返回 false | 记录日志，退出程序 |
| 读取超时 | 500ms 内无数据 | 跳过本次采集 |
| 写入失败 | `write()` 返回 -1 | 记录日志，跳过本次采集 |
| 串口断开 | 读写异常 | 尝试重新打开串口 |
